# DCO 인프라 용어 사전

### 2. 내가 만든 DCO 인프라 용어 정의

**Rack** — 서버들을 세로로 층층이 꽂아 두는 철제 캐비닛으로, 데이터센터에서 장비 위치를 말할 때 기준이 되는 단위다.

**Server** — 랙에 꽂혀 24시간 쉬지 않고 돌아가는 업무용 컴퓨터로, 점검·부품 교체 등 대부분의 작업 대상이 된다.

**NIC** — 서버가 네트워크에 연결되도록 해 주는 통신 부품으로, 이게 문제면 서버가 멀쩡해도 외부와 통신이 안 될 수 있다.

**ToR Switch** — 랙 위쪽에 설치되어 그 랙의 서버들을 묶어 바깥 네트워크로 연결하는 장비로, 문제가 생기면 랙 전체가 영향을 받을 수 있다.

**Link Down** — 네트워크 연결 자체가 끊어졌다는 신호다. 원인은 케이블, 부품, 반대편 장비 등 여러 가지일 수 있어 단정할 수 없다.

**CRC error** — 데이터가 도착은 했지만 검사해 보니 내용이 깨져 있다는 신호로, 연결은 살아 있지만 품질에 문제가 있다는 뜻이다. 전송 경로의 물리적 문제인 경우가 많다고 알려져 있지만 설정 문제 등 다른 원인도 있을 수 있다.

**SSD failure** — 서버의 저장 부품이 고장 난 상태로, 부품 교체 작업으로 이어지는 흔한 고장 유형이다.

**Fan Alert** — 서버 내부 냉각 팬에 이상이 있다는 경고로, 방치하면 온도 상승으로 번질 수 있는 예고 신호다.

### 3. 헷갈리는 용어 비교

#### 비교 1

**내가 비교한 용어:** Link Down vs CRC error

**내가 이해한 차이:** Link Down은 연결 통로 자체가 끊어진 것이고, CRC error는 통로는 살아 있지만 도착한 데이터가 깨져 있는 것이다. "Down은 끊김, CRC는 깨짐"으로 구분한다.

#### 비교 2

**내가 비교한 용어:** Packet Loss(패킷 유실) vs CRC error

**내가 이해한 차이:** 패킷 유실은 데이터가 아예 도착하지 않은 것이고, CRC error는 도착은 했지만 검사해 보니 내용이 깨져 있는 것이다. 처음에는 둘 다 "네트워크 문제"로 묶어서 생각했는데, 서로 다른 현상이었다. 유실은 네트워크 혼잡 등으로 장비가 패킷을 버리는 경우처럼 물리 고장이 아닌 원인도 흔한 반면, CRC 오류는 "전송 중에 데이터가 실제로 변형됐다"는 증거라서 신호가 지나가는 물리 구간(케이블, 커넥터 등)을 먼저 확인하는 단서로 쓰이는 경우가 많다고 알려져 있다. 다만 설정 불일치 같은 다른 원인도 있을 수 있어 단정할 수는 없다.

**기억법:** 안 온 것(유실) / 안 이어진 것(Link Down) / 깨져서 온 것(CRC) — 세 가지는 서로 다른 현상이다.

### 4. 앞으로 사용할 학습 보조 프롬프트

#### 프롬프트 1. 모르는 DCO 용어 설명용

- 역할: DCO 직무를 처음 배우는 학생을 돕는 학습 도우미
- 목표: [용어]를 비전공자도 이해하게 설명. 나는 읽고 내 말로 다시 정의할 예정
- 상황: AWS DCO 인턴십 직무 이해 수업 중. 공개 직무 맥락과 교육용 예시로만 설명
- 출력 형식: (1) 쉬운 설명+비유 1개 (2) 직무에서 나올 수 있는 곳 (3) 연관 용어 1~2개 (4) 바꿔 쓸 때 참고 표현 (5) 초보자가 흔히 오해하는 점
- 제한 조건: 내부 절차 단정 금지 / 실제 장비명·시리얼·티켓 번호·계정·고객정보 생성 금지 / 예시는 SAMPLE·DEMO·EDU 표기와 문서용 IP(192.0.2.x 등)만 / 복사가 아닌 고쳐 쓸 초안 수준으로

#### 프롬프트 2. 비슷한 용어 비교용

- 역할: 헷갈리는 DCO 용어를 비교해 주는 학습 도우미
- 목표: [용어 A] vs [용어 B]의 차이를 구분할 수 있게 설명. 내가 헷갈리는 이유도 함께 입력함
- 상황: 비슷한 용어 구분 연습 중. 공개 직무 맥락 기준
- 출력 형식: (1) A의 쉬운 의미 (2) B의 쉬운 의미 (3) 가장 큰 차이 한 문장+설명 (4) 둘이 같이 등장하는 교육용 예시 (5) 기억법 (6) 단정하면 안 되는 부분
- 제한 조건: 내부 절차·실제 장비명·티켓 번호·고객정보 금지 / 원인·결과 단정 금지 / 교육용 표기만 사용

#### 프롬프트 3. 장애 표현 확인 순서 정리용

- 역할: 장애 신호를 읽고 사고 순서를 연습시키는 DCO 학습 코치
- 목표: 정답 맞히기가 아니라 "단정하지 않고 확인하는 순서" 연습
- 입력: 교육용 장애 표현 (예: "SAMPLE 랙 서버 한 대에서 Link Down 알림 발생")
- 출력 형식: (1) 신호의 의미 한 문장 (2) 가능한 원인 후보 3~5개(단정 없이 나열) (3) 확인 순서와 이유(쉽고 영향 적은 것부터) (4) Escalation 고려 조건 (5) 기록해 둘 항목 (6) 흔한 성급한 단정 1가지
- 제한 조건: 실제 SOP처럼 단정 금지, "일반적으로 생각해 볼 수 있다" 수준으로 / 실제 장비·티켓·계정·고객정보 금지 / 교육용 표기와 문서용 IP만 / 실제 환경에서는 해당 조직 절차가 우선함을 답변 끝에 명시

### 5. AI 답변을 보고 내가 수정한 점

- 어려워서 쉽게 바꾼 표현: 예) "체크섬 검증값" 같은 표현을 "보낸 쪽과 받은 쪽의 검사 결과가 다르다"로 풀어 씀
- 실제 정보처럼 보일 수 있어 삭제하거나 수정한 표현: 초안에 단정형 문장이 있는지, "~라고 알려져 있다"가 유지됐는지 확인한 내용
- 너무 단정적이라 바꾼 표현: 예) "CRC 오류는 물리적 문제다"를 "물리 구간 문제인 경우가 많지만 다른 원인도 있다"로 — 이번 대화에서 실제로 다룬 내용입니다
