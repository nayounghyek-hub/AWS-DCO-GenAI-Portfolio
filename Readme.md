# AWS DCO 인턴십 직무 이해를 위한 생성형 AI 활용 및 포트폴리오

## 1. 과정명
* **생성형 AI 활용 및 포트폴리오 제작 과정을 통한 AWS DCO 직무 이해**

## 2. 포트폴리오 목적
본 포트폴리오는 AWS 데이터 센터 운영(DCO, Data Center Operations) 직무의 핵심 업무 메커니즘을 이해하고, 생성형 AI 및 자동화 스크립트를 활용해 인프라 운영 생산성을 높이는 역량을 기르기 위해 작성된 **학습형 포트폴리오**입니다. 
DCO 업무의 기본 용어 이해부터 표준 운영 절차(SOP) 분석, CLI 및 스크립트 기반 인프라 자동화, 장애(Incident) 분석 및 대시보드 시각화까지의 전 과정을 단계별 수업 산출물로 기록하였습니다.

## 3. 교육용 샘플 데이터 사용 안내
* **면책 조항 (Disclaimer):** 본 포트폴리오에 포함된 모든 시나리오, 티켓 데이터, 로그 파일, 인시던트 리포트 및 인프라 구성은 **실제 AWS의 내부 정보나 시스템 데이터가 아닙니다.** 
* 본 작업물은 오직 교육적 목적으로 설계된 **가상의 시나리오와 교육용 샘플 데이터**만을 활용하여 작성되었습니다.

## 4. 각 폴더의 역할
제공된 폴더 구조에 따른 각 단계별 산출물의 역할과 구성은 다음과 같습니다.

* **`01_dco_glossary/` : DCO 핵심 용어 정리**
  * Ticket, SOP, SLA, Escalation, Handover Memo, Incident Report 등 데이터 센터 운영의 근간이 되는 필수 용어와 개념 정의
* **`02_dco_profile/` : DCO 직무 프로필 분석**
  * AWS DCO 엔지니어의 주요 역할, 책임(R&R) 및 핵심 역량 모델링
* **`03_sop_analysis/` : 표준 운영 절차(SOP) 분석**
  * 가상의 SOP 구조를 분석하고, 상황별 적절한 대응 프로세스를 도출하는 프레임워크 학습
* **`04_environment_setup/` : 인프라 및 자동화 환경 구축**
  * AI 툴 및 스크립트 실행을 위한 로컬/클라우드 개발 환경 설정 기록
* **`05_antigravity_cli/` : 가상 CLI 가이드 및 실습**
  * 데이터 센터 인프라 제어를 가정한 가상의 CLI 환경 실습 및 명령어 활용법 정리
* **`06_cli_file_automation/` : CLI 기반 파일 자동화**
  * 반복적인 파일 생성, 이동, 백업 업무를 CLI 명령어로 자동화하는 스크립트 개발
* **`07_log_analysis_script/` : 로그 분석 스크립트 구현**
  * 서버 및 네트워크 장비에서 발생한 가상의 로그 데이터를 파싱하여 이상 징후를 탐지하는 스크립트 작성
* **`08_incident_analysis/` : 인시던트(장애) 시나리오 분석**
  * 데이터 센터 환경에서 발생할 수 있는 가상의 장애 티켓과 상황을 다각도로 분석
* **`09_incident_report/` : 인시던트 리포트 작성**
  * 분석된 장애 시나리오를 바탕으로 발생 원인, 영향도, 조치 내용, 재발 방지 대책을 포함한 전형적인 보고서 서식 작성
* **`10_stitch_dco_dashboard/` : Stitch DCO 대시보드 시각화**
  * 운영 지표 및 티켓 처리 현황(SLA 준수율 등)을 한눈에 모니터링할 수 있는 가상의 대시보드 기획 및 시각화 산출물
* **`11_exam_prep/` : 직무 및 자격시험 준비**
  * DCO 직무 이해도를 점검하고 인프라 관련 지식을 공고히 하기 위한 학습 요약 및 시험 대비 노트

## 10.배운 점

* 생성형 AI를 활용하면 로그 요약, 코드 작성과 문서 초안 작성 시간을 줄일 수 있습니다.
* AI가 빠르게 결과를 생성하더라도 원본 자료와 비교하는 검증 과정이 필요합니다.
* 로그 분석에서는 확인된 사실, 가능한 추정과 미확인 정보를 구분해야 합니다.
* 장애가 복구되었다는 사실만으로 정확한 Root ause가 확인된 것은 아닙니다.
* Severity는 사건의 심각도이고, Priority는 무엇을 먼저 확인할 것인지 정한 순서입니다.
* Ticket 우선순위는 서비스 영향, 반복·지속 여부, 영향 범위, 복구 여부와 현재 상태를 함께 고려해야 합니다.
* Incident Report는 사건의 흐름과 조치 결과를 다른 사람이 이해할 수 있도록 정리해야 합니다.
* Handover Memo는 교대 전에 진행 중인 상황과 다음 확인 사항을 다음 담당자에게 전달하기 위해 작성합니다.
* 기술 결과물뿐 아니라 AI를 어떤 목적으로 사용하고, 무엇을 직접 검증하고 수정했는지를 기록하는 것이 중요합니다.
